## 摘要
检索增强生成（RAG）被广泛部署以提高语言模型的事实准确性，但目前尚不清楚参数量在 7B 或更小的模型是否能有效利用检索到的信息。为了研究这一问题，我们在四种检索条件下（包括无检索、BM25、使用 E5 large v2 的稠密检索以及保证检索段落包含答案的 Oracle 检索），评估了来自 SmolLM2、Qwen2.5 和 Llama 3.1 三个架构系列的五种模型规模（从 360M 到 8B）。我们引入了一种参数化知识划分方法，将模型已经能够回答的问题与需要外部知识的问题区分开来，从而能够将利用失败与检索质量失败隔离开来。我们发现了三个主要结果：首先，即使在 Oracle 检索下，对于模型自身无法回答的问题，7B 或更小规模的模型在 85% 到 100% 的情况下无法提取出正确答案，这表明存在根本性的利用瓶颈。其次，添加检索上下文会破坏模型先前已知答案的 42% 到 100%，这表明存在一种由上下文存在而非其质量驱动的干扰效应。第三，对 2588 次 Oracle 失败的错误分析显示，主要的失败模式是无关生成，即模型完全忽略了提供的上下文。这些模式在多种提示词模板和检索方法中均成立。结果表明，对于参数量低于 7B 的模型，RAG 的主要限制在于上下文利用而非检索质量，并且在标准评估条件下，在此规模部署 RAG 可能会导致净负收益的权衡。

## Abstract
Retrieval augmented generation RAG is widely deployed to improve factual accuracy in language models yet it remains unclear whether smaller models of size 7B parameters or less can effectively utilize retrieved information. To investigate this question we evaluate five model sizes from 360M to 8B across three architecture families SmolLM2 Qwen2.5 and Llama 3.1 under four retrieval conditions including no retrieval BM25 dense retrieval using E5 large v2 and oracle retrieval where the retrieved passage is guaranteed to contain the answer. We introduce a parametric knowledge split that separates questions a model can already answer from those that require external knowledge which allows us to isolate utilization failure from retrieval quality failure. We find three main results. First even with oracle retrieval models of size 7B or smaller fail to extract the correct answer 85 to 100 percent of the time on questions they cannot answer alone which indicates a fundamental utilization bottleneck. Second adding retrieval context destroys 42 to 100 percent of answers the model previously knew suggesting a distraction effect driven by the presence of context rather than its quality. Third an error analysis of 2588 oracle failures shows that the dominant failure mode is irrelevant generation where the model ignores the provided context entirely. These patterns hold across multiple prompt templates and retrieval methods. The results indicate that for models below 7B parameters the main limitation of RAG is context utilization rather than retrieval quality and that deploying RAG at this scale can lead to a net negative trade off under standard evaluation conditions.

#### 1. 核心问题与整体含义
本文探讨了**检索增强生成（RAG）在小语言模型（SLM，参数量 $\le$ 7B）中的有效性**。尽管 RAG 被广泛用于减少幻觉和更新知识，但现有研究多集中于大模型。作者旨在诊断：对于小模型而言，RAG 的性能瓶颈究竟是**检索质量**（检索不到好内容）还是**上下文利用能力**（检索到了但不会用）。

#### 2. 核心方法论
研究采用了一种**诊断性实验设计**，核心思想是通过消除检索阶段的不确定性来孤立观察模型的利用行为：
*   **Oracle 检索（金标准检索）**：从相同的 50 万个维基百科段落中，人工确保检索到的第一条结果必然包含正确答案。这消除了检索器的误差，直接测试模型的“提取”能力。
*   **参数化知识拆分（Parametric Knowledge Split）**：
    *   **KNOWN（已知）**：模型在无检索情况下能答对的问题。
    *   **UNKNOWN（未知）**：模型在无检索情况下答错的问题。
    *   通过这种拆分，可以分别衡量检索带来的“收益”（解决未知问题）和“损害”（干扰已知知识）。
*   **错误分类法**：将 Oracle 失败案例分为无关生成、提取错误、错误实体、部分匹配、冗余正确和拒绝回答六类。

#### 3. 实验设计
*   **模型选择**：涵盖 360M 到 8B 参数的五个模型，包括 SmolLM2-360M、Qwen2.5 (1.5B, 3B, 7B) 和 Llama-3.1-8B。
*   **数据集**：Natural Questions (NQ，单跳事实) 和 HotpotQA (多跳推理)，共 1,000 个评估问题。
*   **对比方法**：
    *   **检索条件**：无检索 (None)、BM25、稠密检索 (E5-large-v2)、混合检索 (RRF) 和 Oracle 检索。
    *   **提示词消融**：强制性 (Forced)、许可性 (Permissive) 和最小化 (Minimal) 提示词。
*   **基准指标**：Exact Match (EM) 和 F1 分数。

#### 4. 资源与算力
*   **算力平台**：实验在 **Kaggle 免费层 T4 GPU** 实例上完成。
*   **技术细节**：本地模型使用了 4-bit NF4 量化（通过 BitsAndBytes），Llama-3.1-8B 则通过 Groq API 使用 FP16 精度进行验证。
*   **延迟数据**：文中提供了详细的检索延迟分析（如 BM25 约 3.5ms，稠密检索约 230ms）。

#### 5. 实验数量与充分性
*   **实验规模**：进行了 1,000 个问题的多轮测试，涵盖 5 种模型规模和 4 种检索环境。
*   **充分性**：作者不仅做了主实验，还进行了**提示词鲁棒性测试**、**跨架构验证**（Qwen vs Llama）以及**详细的错误分类分析**（分析了 2,588 个失败案例）。
*   **客观性**：通过 Oracle 检索隔离了变量，并使用 McNemar 检验和 Bonferroni 校正确保统计显著性，实验设计非常严谨且具有说服力。

#### 6. 主要结论与发现
*   **利用瓶颈显著**：即使在 Oracle 检索下，7B 模型对 UNKNOWN 问题的提取率也仅为 **14.6%**，这意味着 85% 以上的完美检索被浪费了。
*   **严重的干扰效应**：引入检索上下文会破坏模型原有的记忆。对于 KNOWN 问题，检索导致 7B 模型丢失了 **41.6% - 51.2%** 的正确答案。
*   **存在即干扰**：对于 <7B 的模型，Oracle 检索和噪声检索造成的干扰程度相似，说明**上下文的存在本身（而非质量）**是导致小模型“遗忘”已知知识的主因。
*   **主导失败模式**：61% - 100% 的失败源于**无关生成**，即模型完全无视上下文，按自己的逻辑胡乱回答。
*   **净收益为负**：在标准 RAG 部署下，由于对已知知识的破坏超过了对未知知识的补充，小模型的整体准确率往往不升反降。

#### 7. 优点
*   **变量隔离精准**：通过 Oracle 检索和参数化拆分，清晰地揭示了“利用能力”才是 SLM 的核心短板。
*   **实战指导性强**：量化了 RAG 在小模型上的“负收益”风险，提醒开发者不要盲目在边缘设备的小模型上堆砌检索系统。
*   **分类细致**：错误分类法深入探讨了模型为何失败（如 7B 模型的高拒绝率），而非仅仅给出分数。

#### 8. 不足与局限
*   **任务局限性**：研究仅限于**抽取式问答**，未探讨摘要、对话或生成式推理任务。
*   **语料库规模**：使用的维基百科子集仅占全量的 2%，虽然保证了 Oracle 覆盖，但可能无法完全模拟超大规模知识库的复杂性。
*   **量化影响**：虽然用 FP16 做了验证，但大部分实验基于 4-bit 量化，极小模型（如 360M）的性能可能受量化影响较大。
*   **模型架构单一**：主要测试了 Decoder-only 架构，未涉及 Encoder-Decoder 模型（如 T5 系列）在 RAG 中的表现。
